### Changes Since Previous Runs

Pipeline changes between old and new runs:
- Default Ollama context bumped from 8K to 16K tokens
- Class/ancestry locked in JSON schema to prevent identity drift
- Class-granted skills now injected as mandatory (prevents untrained-skill errors for class skills)
- Dedication instructions added to plan prompt
- `ollama_options` passthrough (used by mistral-small-tuned)

**Validity improved** from ~28% average to 42%. Theme and synergy scores also improved across the board (~+1 on theme, +0.3 on synergy). The pipeline fixes (schema locking, class skill injection) are helping.

**mistral-small-tuned** achieved the best validity (60%) with tuning params (lower temp, repeat penalty, min_p, 32K context) — a notable improvement over base mistral-small (40%).

**mistral-with-ranking** dropped from 24% → 20% validity. Vector ranking boosts theme scores (8.6, highest) but introduces more duplicate feats, suggesting the ranking signal may be too strong and causing the model to fixate.

**qwen3-with-ranking** is the slowest (213s avg, up to 375s) due to thinking overhead but achieved 50% validity — best among the thinking+ranking configs.

## Probable Causes for Invalid Builds

### 1. Skill Prerequisites (50 errors — 39% of all errors)
The single largest error category. Models select feats that require trained/expert proficiency in skills but don't assign those skills. Most common culprits:
- **Arcana/Nature/Occultism/Religion** — caster feats almost always require training in these, but models skip skill assignment
- **Athletics/Warfare Lore** — frontline builds frequently pick feats needing these
- **Crafting** — Firework Technician and similar require Crafting training

**Root cause**: The pipeline doesn't expose which skills the character has trained when the model selects feats. The model has no feedback loop to cross-check feat prerequisites against skill assignments.

**Fix direction**: Include trained skills in the generation context, or add a skill-validation pass before final output.

### 2. Duplicate Feats (29 errors — 22%)
Models pick the same feat in multiple slots. Especially bad with vector ranking enabled (mistral-with-ranking), where ranked feats get repeated. Also affects qwen3-with-ranking on exemplar-thrown and fire-caster builds.

**Root cause**: The duplicate-removal logic catches some cases on repair, but the model's initial plan often contains duplicates that survive through the pipeline. Vector ranking makes this worse by heavily promoting a small set of "best" feats.

**Fix direction**: Deduplicate the feat plan before passing to full generation. Consider de-ranking already-selected feats in vector results.

### 3. Dedication Ordering (18 errors — 14%)
Taking a second Dedication feat before having 2+ non-dedication archetype feats from the first. This is a PF2e multiclass rule that models consistently violate.

**Root cause**: The plan prompt now mentions dedication ordering, but models don't count archetype feats from each dedication before adding another. The constraint is structural (slot-position-dependent) and hard for models to reason about in a single pass.

**Fix direction**: Add a dedicated validation pass that checks dedication ordering in the plan phase, or enforce it structurally in the schema (e.g., mark which slots can contain dedications).

### 4. Missing Prerequisite Feats (8 errors)
Feats that require other feats (e.g., "Adaptive Adept" requires "Adapted Cantrip") but the prerequisite was never taken.

**Root cause**: Feat prerequisite chains are not surfaced to the model during generation. The model sees feat names but not their dependency trees.

### 5. Ability Score Prerequisites (7 errors)
Feats requiring STR/DEX +2 or +3 when the character's stats don't meet the threshold.

**Root cause**: Ability scores are allocated in the skeleton pass but not cross-referenced when selecting feats. The model picks feats that sound thematically right without checking stat requirements.

## Key Takeaways

1. **Validity is improving** (28% → 42%) but still far from production-ready. The 16K context, schema locking, and class skill injection all helped.
2. **Skill prerequisites are the #1 blocker** — 39% of all errors. This is the highest-leverage fix target.
3. **Tuning helps**: mistral-small-tuned (60% valid) outperforms base mistral-small (40%) with just parameter adjustments.
4. **Vector ranking hurts validity** while boosting theme scores. It causes more duplicate feats and needs dedup improvements.
5. **Hard cases remain hard**: exemplar-thrown, fire-caster, and frontline-controller fail across all models. These need pipeline-level fixes, not better models.
6. **qwen3 is slow but capable** (50% valid with ranking) — worth the extra time for quality, but 213s avg may be impractical for interactive use.
